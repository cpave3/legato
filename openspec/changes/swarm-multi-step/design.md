## Context

Legato already has a fully wired swarm system: conductor-driven planning, worker dispatch via `ApplyApprovedPlan`, progress/question/built event bus, per-subtask agent sessions, and a TUI coordination panel. The missing piece is the ability to group subtasks into ordered *steps* where step N cannot begin until step N−1 is terminal.

Currently:
- `Plan.Subtasks` is a flat list.
- `ApplyApprovedPlan` creates all rows at once.
- `Dispatch` fires for any queued subtask regardless of its peers.
- `maybeNotifyAllIdle` only knows about the whole swarm, not per-step completion.

## Goals / Non-Goals

**Goals:**
- Allow a swarm plan to declare 1+ steps, each with its own sub-task list.
- Gate dispatch so step N sub-tasks are only dispatchable after step N−1 is entirely terminal.
- Notify the conductor when a step completes so it can approve advancing to the next step.
- Show step grouping in the TUI swarm panel.
- Provide a CLI verb and API endpoint to advance to the next step.
- Keep backward compatibility: a plan with one step behaves like today's single-shot swarm.

**Non-Goals:**
- Automatic advancement (the conductor or user must explicitly confirm `next-step`).
- Parallel step execution (steps are strictly sequential).
- Dynamic plan mutation after approval (not supported today and not changed).
- Changes to the subtask lifecycle states (queued → dispatched → in_progress → reporting → done/cancelled).

## Decisions

### 1. YAML shape: `steps:` array containing `name` + `subtasks`
- **Rationale**: Clear scoping, self-describing, trivial to distinguish single-step vs multi-step plans.
- **Alternative considered**: Adding a `step` field on each `PlanSubtask`. Rejected because it makes the plan harder to read and doesn't enforce step grouping at the schema level.

### 2. `step_index` stored on `swarm_subtasks` row
- **Rationale**: The simplest way to make the database step-aware. Queries like "all subtasks for step 1" are trivial and indexed.
- **Alternative considered**: A separate `swarm_steps` table. Rejected because it adds a join for every subtask list query with no benefit to the current feature set.

### 3. Step advancement is explicit (`legato swarm next-step <parent-id>`)
- **Rationale**: Prevents the conductor from accidentally spawning workers before verifying step N−1 results. Matches the human-in-the-loop philosophy already present in `RequireUserClose`.
- **Alternative considered**: Auto-advance when `all_idle` fires. Rejected because it removes the conductor's checkpoint.

### 4. No changes to `MaxConcurrentAgents` semantics
- **Rationale**: The cap governs *live workers*, not *steps*. A step might have 20 sub-tasks but only 5 live at once. This is orthogonal and should not be conflated.

### 5. `maybeNotifyAllIdle` gains a `step_completed` overload
- **Rationale**: We already have a per-parent idle check. Extending it to also check per-step keeps the conductor notification pattern consistent (send-keys pointer + inbox event).

## Risks / Trade-offs

- **[Risk]** Existing plan YAMLs break because `subtasks:` moves inside `steps:`.  
  → **Mitigation**: Provide a clear error from `ValidatePlan` telling the user to wrap their subtasks in a `steps:` array. Most plans are generated by a conductor, not hand-written, so the impact is low.
- **[Risk]** TUI performance if a parent has many steps with many subtasks.  
  → **Mitigation**: `ListSubtasksByParent` remains a single indexed query; grouping is done in-memory in the service layer.
- **[Risk]** A conductor could call `next-step` before the current step is actually terminal (if it bypasses `all_idle`).  
  → **Mitigation**: `NextStep` in the service layer validates the previous step is terminal before allowing advancement. Returns an error if not.

## Migration Plan

1. Add migration `016_swarm_step_index.sql` — `ALTER TABLE swarm_subtasks ADD COLUMN step_index INTEGER NOT NULL DEFAULT 0`.
2. Update `ValidatePlan` to accept the new `steps:` shape.
3. Update `ApplyApprovedPlan` to write `step_index`.
4. Update `Dispatch` to check the current active step.
5. Add `NextStep` service method + HTTP handler + CLI verb.
6. Update TUI swarm panel to group by step and show current step.
7. Update plan tests.

## Open Questions

- Should the TUI allow the *user* (not just the conductor) to click a button to advance to the next step? For now, yes — the UI action calls the same `NextStep` API.
- Should the conductor's `brief` include step descriptions? Defer to prompt engineering; the prompt field on each subtask already allows per-task context.
